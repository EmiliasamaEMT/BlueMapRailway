## 2026-05-21 M1 重构推进

### 更新内容

- 建立 `core`、`paper`、`fabric` 三个 Gradle 模块，根项目改为多模块构建。
- 现有 Paper 插件源码与资源迁移到 `paper/` 模块目录，保留当前 Paper 版本为正式可用实现。
- 将首批纯核心模型与算法迁移到 `core/`，包括铁路模型、route/station/edit 基础对象、线路图构建与过滤逻辑。
- 去除 `RailType`、`RailNode`、`RailPosition` 中对 Bukkit `Material`、`Rail.Shape`、`World` 辅助方法的直接依赖，为后续 Fabric 适配做准备。
- 在 `core/` 新增平台边界接口骨架，包括日志、路径与调度抽象。
- 在 `core/` 新增核心配置 DTO，并让 `paper` 侧通过配置翻译器提供线路过滤与 route 自动延续所需配置。
- 保持 `gradlew build` 在多模块结构下继续通过，并恢复根目录 `build/libs/` 自动同步 `paper` 模块的发布 jar，兼容既有 CI 与 Release 流程。

### 技术决策

- M1 阶段优先迁移“纯模型 + 纯算法 + 配置 DTO”，暂不强行把所有注册表与文件读写逻辑一次性搬入 `core`，避免在尚未建立稳定边界时拆坏 Paper 主干。
- 继续保留 `paper` 作为当前唯一正式发布目标；`fabric` 先建立骨架，不在 M1 引入未验证的运行时代码。
- 根目录保留统一构建入口，并通过复制 `paper` 产物回 `build/libs/` 的方式兼容现有 GitHub Actions 与发布脚本。

- 已恢复根目录 `build/libs/BlueMapRailway-0.1.16-SNAPSHOT.jar` 同步输出。

### 下一步计划

- 继续拆薄 `RailwayService`，将扫描协调、状态组织与文件读写门面进一步分层。

## 2026-05-20 0.1.14

### 更新内容

- 增加定期备份服务，默认每 24 小时自动备份 `routes.yml`、`stations.yml`、`edits.yml` 和可选的 `config.yml`。
- 备份采用 zip 单文件归档，默认写入 `plugins/BlueMapRailway/backups/`，便于拷贝、恢复和长期留档。
- 增加 `backup.enabled`、`backup.interval-hours`、`backup.directory`、`backup.include-config`、`backup.max-files` 默认配置，并支持升级后自动补齐缺失项。
- 增加 `/railmap backup`，用于立即手动生成一份备份，不受自动备份周期限制。
- 将 `backup.max-files` 默认值调整为 `0`，即默认无限保留备份，并在文档中明确说明 `0` 的语义。
- 管理网页新增 `edits.yml` 裁切规则层，可保存“世界 + box 区域”形式的线路裁切规则。
- 重扫后会继续应用这些裁切规则，机器里的误识别铁轨不会重新显示出来。
- 管理网页新增“框选线路”模式，可直接框选命中的可见 component 用于线路归类。
- 管理网页新增“线路裁切”模式，可直接在地图上框选裁切区域并保存。
- 管理网页新增“点击时自动载入已有线路”和“框选吸附到整格”两个开关。
- 管理网页新增裁切规则列表，可编辑、删除、定位查看已有规则。
- 管理网页默认改为只读浏览模式，只有输入密钥后才切换到完整管理模式。

### 技术决策

- 备份只覆盖体量较小且人工维护价值较高的 YAML 数据文件，不备份 `cache/rail-cache.yml`，避免把可重建缓存一并长期归档。
- 定时任务采用“周期检查 + 到期才备份”的策略，这样既能跨重启延续 24 小时周期，也不会因为服务器重启而重复生成大量无意义备份。
- 第一版线路修剪通过 `RailEditRegistry` 在扫描结果之后、线路归类之前应用，不改动底层扫描器的图构建逻辑。
- 裁切规则当前采用 box 方式裁掉区域内线段，跨越区域的折线会被拆成区域外可见的小段；这能处理大多数误识别和人工截断场景。
- 这一版不重建新的 component ID，component 仍沿用扫描阶段的原始连通分量，只在显示、SVG 导出和管理网页可见结果上应用裁切。

- 已通过 `gradlew build`。
- 已通过 `node --check src/main/resources/web/app.js`。

### 下一步计划

- 视真实服务器数据量，评估是否需要再增加“手动立即备份”和“导出/恢复指定备份包”能力。

## 2026-05-12 0.1.5

### 更新内容

- 增加历史扫描缓存 `RailChunkCache`。
- 默认保存已扫描且包含铁轨的区块到 `plugins/BlueMapRailway/cache/rail-cache.yml`。
- 扫描开始时自动读取缓存，并与当前已加载区块扫描结果合并。
- 当前已加载区块会覆盖对应缓存；如果铁轨被拆除，该区块缓存会被清除。
- 增加 `ChunkLoadEvent` 监听，玩家或服务器加载新 chunk 后会自动加入区块扫描队列。
- 增加定点区块扫描模式，用历史缓存 + 新加载区块结果重新生成 BlueMap 覆盖层和 SVG。
- 增加 `cache.scan-newly-loaded-chunks` 和 `cache.chunk-load-debounce-ticks` 配置项。
- 增加 `docs/未来展望-Fabric支持.md`，记录 Fabric 1.21.11 支持的模块化改造方向。
- `/railmap status` 和 `/railmap debug` 增加缓存区块、缓存铁轨统计。
- 更新 README、使用文档和技术设计文档，说明缓存的自动执行和边界。

### 技术决策

- 缓存只保存有铁轨的区块，避免大量空区块膨胀缓存文件。
- 第一版缓存使用 YAML 文件，便于服主查看和排查。
- 缓存不主动加载区块，不读取 `.mca`，优先避免卡服风险。
- `ChunkLoadEvent` 只触发对应 chunk 的延迟扫描，不触发完整重扫，避免玩家移动时反复扫描所有已加载区块。
- 完整重扫会取消尚未执行的 chunk-load 局部扫描队列，避免重复扫描。
