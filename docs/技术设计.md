# 技术设计

本文档记录 BlueMapRailway 的实现方案。随着版本迭代，新增模块、关键决策和风险都应继续补充到这里。

## 目标

BlueMapRailway 的目标是在 Paper 26.1.2 服务端自动维护一个 BlueMap 铁路覆盖层，用于显示原版 Minecraft 的 4 种铁轨：

- `RAIL`
- `POWERED_RAIL`
- `DETECTOR_RAIL`
- `ACTIVATOR_RAIL`

插件不修改 BlueMap 渲染器本体，而是通过 BlueMap API 写入 `MarkerSet` 和 `LineMarker`。

## 当前架构

```text
BlueMapRailwayPlugin
  -> RailwayService
      -> RailScanner
          -> RailChunkCache
          -> RailBlockReader
          -> RailGraphBuilder
      -> RailRouteRegistry
      -> RailStationRegistry
      -> BlueMapRailRenderer
      -> SvgRailExporter
  -> RailwayBlockListener
```

### BlueMapRailwayPlugin

主插件类，负责：

- 保存默认配置。
- 注册 `RailwayBlockListener`。
- 注册 BlueMap API 生命周期回调。
- 处理 `/railmap` 管理命令。

### RailwayService

服务调度层，负责：

- 保存当前 `BlueMapAPI` 实例。
- 管理完整扫描、区块扫描和延迟重扫任务。
- 调用 `RailScanner` 分批扫描。
- 调用 `RailRouteRegistry` 应用线路命名和配色。
- 调用 `RailStationRegistry` 读取站点区域。
- 调用 `BlueMapRailRenderer` 写入 BlueMap。
- 调用 `SvgRailExporter` 导出地理型 SVG。
- 提供状态文本给管理员命令。

### RailBlockReader

铁轨识别层，输入 Bukkit 方块数据：

```text
World + x/y/z + Material + BlockData
```

输出内部 `RailNode`。它会读取：

- 铁轨类型：`RailType`
- 坐标：`RailPosition`
- 形态：`Rail.Shape`
- 是否通电：`Powerable#isPowered`
- 连接方向：`RailDirection`

### RailGraphBuilder

图构建层，输入 `Map<RailPosition, RailNode>`，输出 `RailGraphResult`。

当前策略：

- 根据 `Rail.Shape` 计算每个节点的连接方向。
- 生成无向邻接表，并要求连接关系通过双向兼容检查。
- 显式生成 `RailComponent`。
- 为每个 component 生成稳定 ID。
- 从端点或分叉点开始行走。
- 将连续、同类型、同通电状态的铁轨合并成一条 `RailLine`。
- 对纯环路做第二轮兜底遍历。
- 根据 `RailLineFilter` 过滤疑似废弃矿井线段。

连接检查说明：

- 普通平面连接要求相邻铁轨也能指回当前铁轨。
- 上坡连接允许下方坡轨指向上方铁轨，同时上方铁轨在水平方向上具有回连方向。
- 这样可以减少复杂线路附近“单向误连”的概率，同时保留常见坡道。

component ID 生成依据：

- 世界名。
- 节点数量。
- 估算长度。
- 边界盒。
- 排序后的采样节点。

当前实现仍是 MVP 级别。后续需要继续在真实服务器里验证复杂坡道、道岔和跨 chunk 边界。

### RailRouteRegistry

线路配置层，读取插件数据目录中的：

```text
plugins/BlueMapRailway/routes.yml
```

示例：

```yaml
version: 1

routes:
  main-line:
    name: "主线"
    color: "#22c55e"
    line-width: 6
    components:
      - "world:component:8e5f3a12abcd"
```

`RailRouteRegistry` 会在扫描完成后按 component ID 为 `RailLine` 应用：

- route ID
- route 显示名称
- route 颜色
- route 线宽

该设计让扫描层保持纯物理识别，线路命名与配色作为后处理应用。

### RailStationRegistry

站点配置层，读取插件数据目录中的：

```text
plugins/BlueMapRailway/stations.yml
```

示例：

```yaml
version: 1

stations:
  spawn:
    name: "出生点站"
    world: "world"
    area:
      type: box
      min: [120, 60, -30]
      max: [170, 75, 20]
```

当前第一版只使用 box 区域的 `min` 和 `max` 计算站点中心点。站点不会改变铁路图构建结果，只作为 BlueMap POI marker 和 SVG 标注输出。

### RailLineFilter

过滤层用于减少零碎铁轨和废弃矿井铁轨对 BlueMap 的干扰。

全局短线过滤默认规则：

```yaml
filters:
  hide-short-lines: true
  short-line-max-points: 3
  short-line-max-length: 6.0
```

该规则不检查高度和铁轨类型。它按连通分量整体计算点数和估算长度；如果整个连通铁路网络不超过阈值，就直接隐藏。这样可以避免长铁路中因铁轨类型变化切出来的短小 `RailLine` 被误删。它适合过滤零散残留轨道，但阈值过高会误伤短站内支线。

废弃矿井过滤默认规则：

```yaml
filters:
  hide-fragmented-plain-rail-below-min-y: true
  min-y: 50
  fragmented-line-max-points: 8
  fragmented-line-max-length: 32.0
```

一段线会被隐藏，需要同时满足：

- 线段所有点的 Y 坐标都低于 `min-y`。
- 线段自身是普通 `RAIL`。
- 线段所在连通分量中没有动力铁轨、探测铁轨或激活铁轨。
- 线段点数不超过 `fragmented-line-max-points`，或估算长度不超过 `fragmented-line-max-length`。

该规则是启发式过滤，不是结构识别。Minecraft Wiki 只明确普通矿井主要地下生成、恶地矿井可在更高海拔/地表生成，没有提供一个能用于完整过滤的固定 Y 高度。默认 `min-y: 50` 是为了覆盖更多普通地下矿井碎轨，但它不能保证识别所有废弃矿井，也可能过滤短小的地下纯普通铁轨玩家铁路。

### RailScanner

扫描层，当前只扫描配置世界中的已加载区块。

`scan-radius` 规则：

- `-1`：扫描该世界当前所有已加载区块。
- `>= 0`：以世界出生点为中心，扫描半径内已加载区块。

当前版本刻意不主动加载未加载区块，避免服务器启动时造成额外卡顿。

除了完整扫描外，`RailScanner` 也支持输入一组 `ChunkRef` 进行定点区块扫描。该模式用于 `ChunkLoadEvent`：玩家或服务器加载新 chunk 后，`RailwayService` 会把该 chunk 加入待扫描集合；扫描开始时先合并历史缓存，再用新扫描结果覆盖这些 chunk 的缓存。这样可以逐步补全历史铁路数据，而不需要每次都重扫所有已加载区块。

### RailChunkCache

历史扫描缓存层，负责保存已经扫描过且包含铁轨的区块。

默认配置：

```yaml
cache:
  enabled: true
  file: cache/rail-cache.yml
  scan-newly-loaded-chunks: true
  chunk-load-debounce-ticks: 100
```

缓存流程：

1. 扫描开始时读取 `rail-cache.yml`。
2. 只将当前启用世界的缓存节点合并进扫描节点图。
3. 完整扫描会扫描当前已加载区块；区块扫描只扫描由事件传入且仍处于加载状态的 chunk。
4. 被扫描区块使用新结果覆盖对应缓存。
5. 如果该区块已经没有铁轨，则从缓存中移除。
6. 扫描完成后保存缓存文件。

该机制不会主动加载区块，也不会读取 region 文件。它解决的是“插件曾经扫描过的铁路，在区块卸载后仍能继续显示”的问题。

### BlueMapRailRenderer

BlueMap 输出层，负责：

- 按 `worldName` 分组 `RailLine`。
- 按线路创建 `MarkerSet`。
- 未归类线路进入 `railways.unclassified`。
- 已归类线路进入 `railways.route.<route-id>`。
- 站点进入 `railways.stations`。
- 将每条 `RailLine` 转成 `LineMarker`。
- 将每个 `RailStation` 转成 `POIMarker`。
- 写入该 BlueMap world 下的所有 `BlueMapMap`。

渲染配置包括：

- MarkerSet ID
- MarkerSet 标签
- 默认隐藏状态
- 线宽
- Y 轴抬高量
- 深度测试开关
- 各类铁轨颜色

### SvgRailExporter

SVG 导出层，负责把扫描结果中的 `RailLine` 输出为一个静态 SVG 文件。

当前策略：

- 使用地理型投影，不做地铁线路图式自动布局。
- 将 Minecraft `x` 坐标映射为 SVG `x`。
- 将 Minecraft `z` 坐标映射为 SVG `y`。
- 忽略高度 `y` 对二维布局的影响。
- 每条 `RailLine` 输出为一个 `<polyline>`。
- 按世界名创建 `<g>` 分组。
- 按 route 创建 SVG `<g>` 分组。
- 每条线保留 `data-world`、`data-component-id`、`data-type`、`data-powered` 属性，方便网页端二次处理。
- 已归类线路额外保留 `data-route-id` 和 `data-route-name`。
- 站点输出为 SVG `<circle>` 和 `<text>`。
- 默认使用与 BlueMap 覆盖层一致的颜色和线宽。

默认输出：

```text
plugins/BlueMapRailway/export/railways.svg
```

配置：

```yaml
export:
  svg:
    enabled: true
    directory: export
    file: railways.svg
    title: BlueMap Railway
    padding: 16.0
    non-scaling-stroke: true
```

`non-scaling-stroke` 默认开启，使网页缩放 SVG 时线条宽度保持稳定，更适合嵌入网页查看。

## 线程模型

当前扫描任务运行在 Bukkit 主线程，并通过 `chunks-per-tick` 控制每 tick 处理区块数。

这样做的原因：

- Bukkit 世界与方块 API 通常不应在异步线程访问。
- MVP 阶段优先保证正确性和服务端安全。

后续如果要支持全图扫描，应优先考虑 `ChunkSnapshot` 或直接读取 `.mca`，但那会引入额外复杂度。

## Paper 插件形态

Paper 插件本质上是一个 Java `.jar` 文件。jar 内必须包含插件描述文件，本项目使用传统 Bukkit/Paper 兼容的 `plugin.yml`：

```text
src/main/resources/plugin.yml
```

Gradle 构建时会把 `plugin.yml`、`config.yml` 和编译后的 class 一起打包进 jar。服务端启动时，Paper 从 `plugins/` 目录读取 jar，解析 `plugin.yml`，再加载 `main` 指向的主类。

本项目的运行期依赖关系：

- Paper API：仅编译期依赖，由 Paper 服务端提供。
- BlueMap API：仅编译期依赖，由服务器安装的 BlueMap 插件提供。

因此发布 jar 不需要把 Paper 或 BlueMap 打包进去。服务器实际使用时需要同时安装 BlueMap。

## Release 发布流程

项目包含两个 GitHub Actions：

- `.github/workflows/build.yml`：推送到 `main` 或创建 PR 时执行构建检查，并上传开发构建产物。
- `.github/workflows/release.yml`：推送 `v*` tag 时构建正式 jar 并创建 GitHub Release。

版本号由 Gradle 属性 `pluginVersion` 控制：

```bash
./gradlew build -PpluginVersion=0.1.0
```

本地不传该属性时，默认使用当前开发快照版本，例如 `0.1.5-SNAPSHOT`。

正式发布步骤：

```bash
git tag v0.1.0
git push origin v0.1.0
```

GitHub Actions 会从 tag 中取出 `0.1.0`，构建：

```text
build/libs/BlueMapRailway-0.1.0.jar
```

然后把这个 jar 附加到 GitHub Release。

## 自动更新模型

当前监听以下事件：

- `BlockPlaceEvent`
- `BlockBreakEvent`
- `BlockPhysicsEvent`
- `BlockRedstoneEvent`
- `ChunkLoadEvent`

当事件涉及铁轨时，不立即扫描，而是根据 `scanner.update-debounce-ticks` 延迟触发完整重扫。短时间内的多次变化会被合并。

`ChunkLoadEvent` 不会触发完整重扫。默认配置下，它只把新加载的 chunk 记录到 `RailwayService` 的区块扫描队列，并根据 `cache.chunk-load-debounce-ticks` 合并短时间内的大量加载事件。区块扫描完成后会重新生成 BlueMap Marker 和 SVG，因此玩家经过旧铁路区域后，历史缓存会逐步补全。

后续计划继续把铁轨放置、破坏等事件也改成 chunk/region 级 dirty 标记，只重算受影响区域。

## 已知限制

- 只扫描已加载区块，不会主动发现离线区域中的铁轨。
- 历史缓存只能保留插件曾经扫描过的区块；从未被插件扫描过的旧铁路需要等对应 chunk 被再次加载，或未来通过离线扫描补齐。
- 事件更新目前触发完整重扫，不是局部增量。
- 斜坡连接只做基础支持，复杂高度关系需要在真实服务器里验证。
- 缓存当前使用单个 YAML 文件，超大地图可能需要后续拆分为按世界或 region 存储。
- 线路命名和配色目前依赖 component ID，线路大规模改造后可能需要重新绑定。
- 站点区域目前只用于显示 marker，不参与线路识别切分。
- 没有双轨走廊等更高级铁路网络功能。
- SVG 当前是地理型导出，不会自动生成规整的地铁线路示意图。

## 下一阶段方向

- 使用 `ChunkSnapshot` 降低主线程扫描压力。
- 将铁轨变化事件也改为 chunk 级 dirty 更新。
- 评估按世界或按 region 拆分缓存文件。
- 支持管理员配置不同世界的扫描范围。
- 增加站点 marker 和线路命名。
