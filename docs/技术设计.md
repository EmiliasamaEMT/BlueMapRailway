# 技术设计

本文档记录 BlueMapRailway 的实现方案。随着版本迭代，新增模块、关键决策和风险都应继续补充到这里。

## 目标

BlueMapRailway 的目标是在 Paper 26.1.2 服务端自动维护一个 BlueMap 铁路覆盖层，用于显示原版 Minecraft 的 4 种铁轨：

- `RAIL`
- `POWERED_RAIL`
- `DETECTOR_RAIL`
- `ACTIVATOR_RAIL`

插件不修改 BlueMap 渲染器本体，而是通过 BlueMap API 写入 `MarkerSet` 和 `LineMarker`。

## 当前架构

```text
BlueMapRailwayPlugin
  -> RailwayService
      -> RailScanner
          -> RailBlockReader
          -> RailGraphBuilder
      -> BlueMapRailRenderer
  -> RailwayBlockListener
```

### BlueMapRailwayPlugin

主插件类，负责：

- 保存默认配置。
- 注册 `RailwayBlockListener`。
- 注册 BlueMap API 生命周期回调。
- 处理 `/railmap` 管理命令。

### RailwayService

服务调度层，负责：

- 保存当前 `BlueMapAPI` 实例。
- 管理扫描任务和延迟重扫任务。
- 调用 `RailScanner` 分批扫描。
- 调用 `BlueMapRailRenderer` 写入 BlueMap。
- 提供状态文本给管理员命令。

### RailBlockReader

铁轨识别层，输入 Bukkit 方块数据：

```text
World + x/y/z + Material + BlockData
```

输出内部 `RailNode`。它会读取：

- 铁轨类型：`RailType`
- 坐标：`RailPosition`
- 形态：`Rail.Shape`
- 是否通电：`Powerable#isPowered`
- 连接方向：`RailDirection`

### RailGraphBuilder

图构建层，输入 `Map<RailPosition, RailNode>`，输出 `List<RailLine>`。

当前策略：

- 根据 `Rail.Shape` 计算每个节点的连接方向。
- 生成无向邻接表。
- 从端点或分叉点开始行走。
- 将连续、同类型、同通电状态的铁轨合并成一条 `RailLine`。
- 对纯环路做第二轮兜底遍历。
- 根据 `RailLineFilter` 过滤疑似废弃矿井线段。

当前实现仍是 MVP 级别。后续需要重点改进斜坡连接、跨 chunk 边界缓存和复杂分叉线路的命名能力。

### RailLineFilter

过滤层用于减少零碎铁轨和废弃矿井铁轨对 BlueMap 的干扰。

全局短线过滤默认规则：

```yaml
filters:
  hide-short-lines: true
  short-line-max-points: 3
  short-line-max-length: 6.0
```

该规则不检查高度和铁轨类型。只要生成出的 `RailLine` 点数不超过阈值，或估算长度不超过阈值，就直接隐藏。它适合过滤零散残留轨道，但阈值过高会误伤短站内支线。

废弃矿井过滤默认规则：

```yaml
filters:
  hide-fragmented-plain-rail-below-min-y: true
  min-y: 50
  fragmented-line-max-points: 8
  fragmented-line-max-length: 32.0
```

一段线会被隐藏，需要同时满足：

- 线段所有点的 Y 坐标都低于 `min-y`。
- 线段自身是普通 `RAIL`。
- 线段所在连通分量中没有动力铁轨、探测铁轨或激活铁轨。
- 线段点数不超过 `fragmented-line-max-points`，或估算长度不超过 `fragmented-line-max-length`。

该规则是启发式过滤，不是结构识别。Minecraft Wiki 只明确普通矿井主要地下生成、恶地矿井可在更高海拔/地表生成，没有提供一个能用于完整过滤的固定 Y 高度。默认 `min-y: 50` 是为了覆盖更多普通地下矿井碎轨，但它不能保证识别所有废弃矿井，也可能过滤短小的地下纯普通铁轨玩家铁路。

### RailScanner

扫描层，当前只扫描配置世界中的已加载区块。

`scan-radius` 规则：

- `-1`：扫描该世界当前所有已加载区块。
- `>= 0`：以世界出生点为中心，扫描半径内已加载区块。

当前版本刻意不主动加载未加载区块，避免服务器启动时造成额外卡顿。

### BlueMapRailRenderer

BlueMap 输出层，负责：

- 按 `worldName` 分组 `RailLine`。
- 创建 `MarkerSet`。
- 将每条 `RailLine` 转成 `LineMarker`。
- 写入该 BlueMap world 下的所有 `BlueMapMap`。

渲染配置包括：

- MarkerSet ID
- MarkerSet 标签
- 默认隐藏状态
- 线宽
- Y 轴抬高量
- 深度测试开关
- 各类铁轨颜色

## 线程模型

当前扫描任务运行在 Bukkit 主线程，并通过 `chunks-per-tick` 控制每 tick 处理区块数。

这样做的原因：

- Bukkit 世界与方块 API 通常不应在异步线程访问。
- MVP 阶段优先保证正确性和服务端安全。

后续如果要支持全图扫描，应优先考虑 `ChunkSnapshot` 或直接读取 `.mca`，但那会引入额外复杂度。

## Paper 插件形态

Paper 插件本质上是一个 Java `.jar` 文件。jar 内必须包含插件描述文件，本项目使用传统 Bukkit/Paper 兼容的 `plugin.yml`：

```text
src/main/resources/plugin.yml
```

Gradle 构建时会把 `plugin.yml`、`config.yml` 和编译后的 class 一起打包进 jar。服务端启动时，Paper 从 `plugins/` 目录读取 jar，解析 `plugin.yml`，再加载 `main` 指向的主类。

本项目的运行期依赖关系：

- Paper API：仅编译期依赖，由 Paper 服务端提供。
- BlueMap API：仅编译期依赖，由服务器安装的 BlueMap 插件提供。

因此发布 jar 不需要把 Paper 或 BlueMap 打包进去。服务器实际使用时需要同时安装 BlueMap。

## Release 发布流程

项目包含两个 GitHub Actions：

- `.github/workflows/build.yml`：推送到 `main` 或创建 PR 时执行构建检查，并上传开发构建产物。
- `.github/workflows/release.yml`：推送 `v*` tag 时构建正式 jar 并创建 GitHub Release。

版本号由 Gradle 属性 `pluginVersion` 控制：

```bash
./gradlew build -PpluginVersion=0.1.0
```

本地不传该属性时，默认版本为 `0.1.0-SNAPSHOT`。

正式发布步骤：

```bash
git tag v0.1.0
git push origin v0.1.0
```

GitHub Actions 会从 tag 中取出 `0.1.0`，构建：

```text
build/libs/BlueMapRailway-0.1.0.jar
```

然后把这个 jar 附加到 GitHub Release。

## 自动更新模型

当前监听以下事件：

- `BlockPlaceEvent`
- `BlockBreakEvent`
- `BlockPhysicsEvent`
- `BlockRedstoneEvent`

当事件涉及铁轨时，不立即扫描，而是根据 `scanner.update-debounce-ticks` 延迟触发完整重扫。短时间内的多次变化会被合并。

后续计划改成 chunk/region 级 dirty 标记，只重算受影响区域。

## 已知限制

- 只扫描已加载区块，不会主动发现离线区域中的铁轨。
- 事件更新目前触发完整重扫，不是局部增量。
- 斜坡连接只做基础支持，复杂高度关系需要在真实服务器里验证。
- 没有持久化缓存，重启后需要重新扫描。
- 没有线路命名、站点、颜色分组等高级铁路网络功能。

## 下一阶段方向

- 使用 `ChunkSnapshot` 降低主线程扫描压力。
- 引入 chunk 级缓存和 dirty 更新。
- 将扫描结果持久化到插件数据目录。
- 支持管理员配置不同世界的扫描范围。
- 增加站点 marker 和线路命名。
