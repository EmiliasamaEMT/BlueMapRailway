# 未来展望：线路分类、命名与配色

本文档记录铁路线路管理能力的设计方向。这里的内容不是当前版本功能，而是后续版本可以逐步实现的规划。

## 目标

当前版本只把扫描到的铁轨渲染成统一的 BlueMap 覆盖层。后续希望支持：

- 将铁路划分成不同线路。
- 给线路命名，例如“主线”“东线”“矿区支线”。
- 为不同线路设置颜色、线宽和显示分组。
- 支持站点、支线、换乘点等更高层语义。
- 在自动扫描更新后尽量保留人工设置。

核心原则：**自动识别只做辅助，最终线路归属应允许管理员手动修正。**

## 概念模型

### RailNetwork

一次扫描得到的完整铁路网络，可以理解为若干连通分量。

```text
RailNetwork
  id
  worldName
  nodes
  edges
  components
```

### RailComponent

一个连通的铁路分量。只要铁轨之间通过形态连接，它们就属于同一个 component。

```text
RailComponent
  id
  worldName
  nodeCount
  estimatedLength
  bounds
  endpoints
  junctions
```

### RailLine

面向 BlueMap 渲染的线段。当前版本已经有 `RailLine`，但它更偏渲染层，可能会因为铁轨类型或通电状态被切成多段。

后续应区分：

- `RailSegment`：图算法切出来的连续片段。
- `RailRoute`：管理员或算法定义的一条业务线路。
- `RenderedRailLine`：最终写给 BlueMap 的线。

### RailRoute

一条有名字、有颜色、有归属规则的线路。

```yaml
routes:
  main-line:
    name: "主线"
    world: "world"
    color: "#22c55e"
    line-width: 6
    visible: true
    selector:
      mode: manual
      components:
        - "world:component:a1b2c3"
```

## 线路识别策略

### 1. 连通分量优先

第一阶段不要试图自动理解“哪条是主线”。先把扫描结果划分为连通分量：

```text
铁轨节点 -> 邻接图 -> 连通分量
```

对于许多服务器，一个地铁系统或铁路系统往往天然是一个大连通分量。这个分量可以先作为一条可命名线路。

优点：

- 实现简单。
- 不容易拆错。
- 人工配置成本低。

缺点：

- 大型交通网会被合成一条线路。
- 分叉、支线、环线无法自动区分。

### 2. 端点与分叉点切段

第二阶段可以按图结构切段：

- 端点：度数为 1 的节点。
- 普通中间点：度数为 2 的节点。
- 分叉点：度数大于 2 的节点。

从端点或分叉点出发，沿度数为 2 的节点行走，直到遇到下一个端点或分叉点，生成一个 `RailSegment`。

这样可以得到：

```text
主线段
支线段
站场短线
环线片段
```

后续管理员可以把多个 segment 合并到一个 route。

### 3. 自动主线推断

可以做启发式，但不建议第一版依赖它：

- 较长的 segment 更可能是主线。
- 连接多个分叉点的 segment 更可能是主干。
- 高比例动力铁轨可能表示玩家修建的交通线。
- 长距离、低曲率、少死端的线路更像主线。

这些规则可以用于默认推荐，而不是直接替代人工配置。

## 站点汇聚处理

站点不应该只被理解成“很多铁轨相连的复杂点”。在服务器铁路系统中，站点经常包含多条股道、道岔、站台线、折返线、支线入口和维护线。如果完全依赖底层铁轨图自动切分，站点区域很容易把多条线路粘成一个巨大的连通分量。

建议把站点作为更高层的语义对象：

```text
物理铁轨 RailNode / RailSegment
        ↓
轨道走廊 TrackCorridor / TrackGroup
        ↓
线路 RailRoute
        ↓
站点 Station
```

核心原则是：**站点是一个区域语义节点，不是单个铁轨交叉点。**

### 站点区域

第一版可以允许管理员手动定义站点区域。区域可以是长方体，也可以是平面半径范围。

示例：

```yaml
stations:
  spawn:
    name: "出生点站"
    world: "world"
    area:
      type: box
      min: [120, 60, -30]
      max: [170, 75, 20]
```

扫描时，落在站点区域内的铁轨可以标记为 `station-internal`。这些铁轨仍然可以显示，但不直接参与普通线路的自动延伸判断。

### 入口与出口

站点区域真正需要暴露给线路系统的是边界上的入口和出口：

```text
线路 A -> 站点入口 -> 站点内部股道 -> 站点出口 -> 线路 A
```

站内有多少道岔不重要，重要的是这条线路从哪个方向进入站点、从哪个方向离开站点。这样可以避免站内复杂结构污染线路识别。

后续可以为站点维护：

- 站点 ID 与显示名称。
- 站点区域。
- 入口/出口方向。
- 关联线路。
- 可选站台编号。

线路配置可以引用站点：

```yaml
routes:
  main-line:
    name: "主线"
    color: "#22c55e"
    stops:
      - spawn
      - east-town
      - mine-base
```

### 自动候选

自动识别站点可以作为后续辅助功能，而不是第一版强依赖。

可用启发式：

- 一定半径内存在多个分叉点。
- 多条长线路在同一区域汇聚。
- 区域内铁轨密度显著高于普通线路。
- 存在多条近似平行的短股道。
- 管理员附近反复执行线路绑定命令的位置。

自动识别结果应作为“候选站点”，由管理员确认后写入 `stations.yml` 或 `routes.yml`。

## 双轨与轨道走廊

常见服务器铁路会使用双轨：两条基本平行的线路，中间隔一格或数格，分别承担不同方向或不同用途。它们在物理图上通常是不连通的两个 component，但在语义上往往属于同一条线路。

因此不要把双轨强行合并到底层铁轨图里。底层仍然保留每一条真实铁轨，上层再把它们归入同一个 `TrackCorridor` 或 `RailRoute`。

```text
左线 component A
右线 component B
        ↓
TrackCorridor: main-line-corridor
        ↓
RailRoute: 主线
```

### 双轨识别启发式

可以用以下条件判断两个 segment/component 是否可能属于同一条双轨走廊：

- 二者距离稳定，例如 2 到 4 格以内。
- 方向基本一致。
- 高度接近。
- XZ 投影有较高重叠比例。
- 连续长度超过阈值，避免把零碎矿井铁轨误判成双轨。
- 两条线之间大部分位置保持近似平行。
- 两条线在相同站点区域附近汇聚或共同通过。

这些规则适合生成建议，不适合直接修改管理员配置。

### 显示模式

双轨线路可以支持两种渲染模式。

真实轨道模式：

```yaml
routes:
  main-line:
    name: "主线"
    color: "#22c55e"
    render-mode: physical-tracks
```

该模式保留两条真实轨道的位置，只是使用同一线路名称和颜色。它更适合 BlueMap 这种空间地图，因为玩家可以看到真实轨道布局。

中心线模式：

```yaml
routes:
  main-line:
    name: "主线"
    color: "#22c55e"
    render-mode: centerline
```

该模式把双轨走廊简化成一条居中的线路，适合铁路网很密、服主更关心线路示意而不是精确轨道位置的场景。

默认建议使用 `physical-tracks`，后续再增加 `centerline` 作为可选模式。

### 与站点的关系

站点和双轨应该互相配合：

- 站外按双轨走廊识别。
- 站内按站点区域处理。
- 线路在站点边界处只关心入口和出口。
- 站内股道可以单独显示，也可以在简洁模式下隐藏。

这样可以避免双轨在站内因为道岔和股道过多而被错误拆分，也避免站点把两条方向轨合并成难以管理的大线团。

## 手动管理方案

### 管理员命令

建议命令形态：

```text
/railmap route list
/railmap route create <id> <名称>
/railmap route color <id> <#RRGGBB>
/railmap route width <id> <宽度>
/railmap route assign-nearest <id> <半径>
/railmap route unassign-nearest <id> <半径>
/railmap route info <id>
```

`assign-nearest` 可以把管理员附近最近的 component 或 segment 归入某条线路。

这比让管理员手写坐标友好很多。

### 交互式选择

后续可以提供两种交互：

1. 管理员站在线路旁，执行命令绑定最近线路。
2. 管理员手持指定物品右键铁轨，将该连通分量或片段加入当前线路。

可选流程：

```text
/railmap route edit main-line
右键铁轨
插件提示：已选择 component xxx，长度 1240，节点 900
/railmap route save
```

## 持久化设计

需要把线路定义保存到插件目录，而不是写进默认 `config.yml`。

建议文件：

```text
plugins/BlueMapRailway/routes.yml
```

示例：

```yaml
version: 1
routes:
  main-line:
    name: "主线"
    world: "world"
    color: "#22c55e"
    line-width: 6
    components:
      - "world:component:8e5f3a"
  mine-branch:
    name: "矿区支线"
    world: "world"
    color: "#f97316"
    line-width: 5
    components:
      - "world:component:91ab20"
```

## 稳定 ID 问题

线路配置要跨重启、跨重扫稳定，核心难点是 component/segment ID。

不能简单用内存对象 ID。推荐生成稳定指纹：

```text
worldName
节点数量
端点坐标
边界盒
若干采样点坐标
```

例如：

```text
world:component:sha1(world + bounds + endpoints + sampledNodes)
```

风险：

- 玩家大规模改造线路后，component ID 可能变化。
- 两个相近短线段可能指纹相似。

缓解方案：

- 保存旧 ID 到新 ID 的迁移候选。
- 如果 ID 找不到，按空间重叠率匹配旧线路。
- 在 `/railmap route status` 中提示“线路绑定丢失”。

当前实现进展：

- `routes.yml` 支持 `anchors`、`bounds` 和 `auto-match`。
- `/railmap route assign-nearest` 会写入 component、锚点和边界盒。
- 扫描后会先按 component ID 精确匹配，失败时再用空间锚点和旧范围尝试自动延续。
- 自动延续只追加新的 component ID，不删除旧 ID；多个 route 竞争同一 component 时只报告冲突，不自动写入。
- 已被全局短线过滤视为零碎线路的 component 不参与自动延续候选，降低误配概率。

## 渲染方案

当前版本所有铁路写入一个 MarkerSet：

```text
Railways
```

线路管理后建议改成：

```text
Railways
Railways / 主线
Railways / 矿区支线
Railways / 未分类
```

BlueMap API 的 MarkerSet 是扁平 ID，不是真正树结构，所以可以用命名约定：

```text
railways.route.main-line
railways.route.mine-branch
railways.unclassified
```

显示标签：

```text
铁路 - 主线
铁路 - 矿区支线
铁路 - 未分类
```

## 未分类线路

任何没有被 route 接管的 component/segment，都进入未分类图层。

配置示例：

```yaml
routes:
  unclassified:
    enabled: true
    color: "#9ca3af"
    default-hidden: false
```

这样管理员可以逐步整理线路，不必一次配置完整。

## 与过滤器的关系

过滤顺序建议：

1. 扫描铁轨节点。
2. 构建完整图和连通分量。
3. 应用全局过滤器，移除零碎/疑似废弃矿井 component。
4. 根据 routes.yml 归类线路。
5. 未匹配部分进入未分类。
6. 输出 BlueMap MarkerSet。

如果一个 component 已经被管理员手动归入 route，应该默认跳过废弃矿井过滤，避免误删人工确认的线路。

建议配置：

```yaml
filters:
  ignore-filtered-if-route-assigned: true
```

## 推荐实现阶段

### 阶段 1：连通分量 ID 与未分类

- 在扫描结果中显式生成 `RailComponent`。
- 每个 component 生成稳定 ID。
- BlueMap marker label 中带上 component ID 的短码。
- 文档说明如何观察 component。

### 阶段 2：routes.yml

- 支持 `routes.yml`。
- 支持按 component ID 指定颜色和名称。
- 未分类线路仍使用默认颜色。

### 阶段 3：管理员命令

- `/railmap route list`
- `/railmap route create`
- `/railmap route color`
- `/railmap route assign-nearest`

当前已补充：

- `/railmap route rename`
- `/railmap route status`
- `/railmap route auto-match`
- `/railmap route anchor-nearest`

### 阶段 4：自动建议

- 扫描完成后对长 component 给出推荐线路。
- 对疑似站场、支线、短线段给出分类建议。
- 对疑似双轨走廊给出合并到同一线路的建议。

### 阶段 5：站点与换乘

- 支持 `/railmap station add <id> <名称> [半径]`。
- 支持手动定义站点区域。
- 站点作为 BlueMap POI marker。
- 支持站点与线路关联。
- 站点区域内铁轨标记为站内轨道，避免干扰普通线路识别。

当前已补充：

- `/railmap station list`
- `/railmap station info`
- `/railmap station add <id> <名称> [半径]`
- `/railmap station set-area-here <id> [半径]`
- `/railmap station remove <id>`
- BlueMap 中已命名线路直接以 route 名称作为独立图层标签。
- 站点 POI 详情会列出穿过站点区域的线路名称或未分类 component 短 ID。
- BlueMap 会绘制站点 box 边框。
- 站点范围内的铁路小段会进入 `站内轨道` 图层，不再混入主线图层。
- 内置管理网页可显示铁路网、点击 component 归类 route、框选站点范围，并写回 `routes.yml` / `stations.yml`。

仍待实现：

- 站点与线路显式关联字段。
- 在扫描阶段引入 station-internal 节点语义，真正让站点区域参与 component 切分和自动延续判断。
- 管理网页支持两点校准、截图上传、多世界背景和 segment 级细分。

### 阶段 6：双轨走廊

- 引入 `TrackCorridor` 或同类概念。
- 支持把多个 component/segment 归入同一线路。
- 默认以真实轨道模式显示双轨。
- 可选支持中心线模式，简化密集铁路网显示。

## 风险与注意事项

- 自动线路分类很容易和服主的命名习惯不一致，应优先支持手动覆盖。
- 大型铁路网可能是一个巨大连通分量，需要 segment 级管理。
- 线路变更后稳定 ID 可能变化，需要迁移或重绑定机制。
- 过多 MarkerSet 会让 BlueMap 菜单变乱，需要可配置是否按线路拆图层。
- 短线过滤和废弃矿井过滤不应删除已人工归类的线路。
- 站点区域会主动改变线路识别边界，需要提供清晰的可视化和诊断信息。
- 双轨自动识别只能作为建议，不能直接覆盖管理员的线路归属。
